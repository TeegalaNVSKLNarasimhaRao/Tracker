Real-Time Architecture
Is data lost on refresh? No. Here is why.
✓ Yes — fully real-time and production-ready

All data is persisted in Firebase Firestore. Nothing is lost on refresh, tab close, or browser restart. The app works offline and syncs when reconnected.

QuestionAnswer
Is data lost on refresh?No — saved to Firestore within 800ms of every action
Data lost on browser close?No — Firestore is a permanent cloud database
Works on multiple devices?Yes — live onSnapshot sync across all tabs/devices
Works offline?Yes — Firestore offline cache; syncs on reconnect
Needs a backend server?No — Firebase is the entire backend
Needs Node.js / Express?No — browser-only JavaScript
Data isolated per user?Yes — Firestore rules enforce UID-scoped access

How the sync cycle works

Sign in with Google
onAuthStateChanged fires → currentUser set → loadFromFirebase() reads all 3 Firestore documents in parallel (meta, tracker, diary).
Any user action (habit tick, water click, diary type…)
saveS(type) is called. It marks the relevant document (meta / tracker / diary) as dirty and starts an 800ms debounce timer. If another action fires within 800ms, the timer resets. After 800ms of silence, all dirty documents are written to Firestore with Promise.all.
Change from another device
The onSnapshot listener on the meta document fires. If habits or holidays changed, the in-memory state updates and refreshView() re-renders the active tab.
Connection lost
Firestore's persistent local cache (IndexedDB) serves all reads and queues writes. When connectivity returns, writes flush automatically.

Firestore data structure

users/ {uid}/ ← your Google UID — completely private data/ meta ← schedule, habits, holidays, focus, water, finance, templates, checkin tracker ← habit ticks ("YYYY-MM-DD_habitId": "done"), notes, work log diary ← diary entries ("YYYY-MM-DD": {text, mood, saved}) profile/ info ← display name, email, lastSeen (for Super Admin panel only)
⚠ Only two things use localStorage (not Firestore)

Widget collapse state (focus/water hide) and dark mode preference are stored in localStorage for instant load-time restore. These are preferences, not data — clearing browser cache resets them only. All actual content lives in Firestore.

User Roles
Three roles, two accounts for the same person
RoleEmailScheduleHabitsAdmin Tab
Owner owner@yourdomain.com Full owner schedule (Big Data, Finance, Work calls) 11 custom habits No
Super Admin admin@yourdomain.com Full owner schedule (same as above) 11 custom habits Yes — 🛡 Admin
General User Any other Google account Clean general template (editable) 7 basic habits No

Role detection logic

function isOwner(){ return currentUser?.email === 'owner@yourdomain.com' // replace with your email; } const SUPER_ADMIN_EMAIL = 'admin@yourdomain.com' // replace with admin email; function isSuperAdmin(){ return currentUser?.email === SUPER_ADMIN_EMAIL; } // Both accounts belong to the same person — both get owner schedule function isOwnerOrAdmin(){ return isOwner() || isSuperAdmin(); } function DEF_WD(){ return isOwnerOrAdmin() ? OWNER_WD : GENERAL_WD; } function DEF_HABITS(){ return isOwnerOrAdmin() ? OWNER_HABITS : GENERAL_HABITS; }
✓ Security: roles cannot be spoofed

Role checks use currentUser.email from Firebase Auth — a verified Google identity that cannot be modified client-side. The same check is enforced server-side in Firestore security rules via request.auth.token.email.

Owner schedule enforcement

Firebase Firestore may contain a stale general schedule from an earlier session. The app enforces the correct owner defaults at two separate points to prevent any race condition:

1. loadFromFirebase()
After loading data, if isOwnerOrAdmin(), all 6 days are force-set from OWNER_WD. Then saveS('meta') is queued 1.5s later to write the correct schedule back to Firestore.
2. applyOwnerSchedule()
Called inside startSync() on every Firestore snapshot. Ensures no subsequent snapshot can overwrite the in-memory owner schedule before the corrected data is saved back.

Required Firestore Security Rules

Go to Firebase Console → Firestore Database → Rules and paste the following, then click Publish:

rules_version = '2'; service cloud.firestore { match /databases/{database}/documents { // Regular users: own data only match /users/{userId}/data/{docId} { allow read, write: if request.auth != null && request.auth.uid == userId && docId in ['meta', 'tracker', 'diary']; } // Profile: user writes own, Super Admin reads/writes all match /users/{userId}/profile/info { allow read, write: if request.auth != null && request.auth.uid == userId; allow read, write: if request.auth != null && request.auth.token.email == 'admin@yourdomain.com'; } // Super Admin: full access to all user data match /users/{userId}/{document=**} { allow read, write, delete: if request.auth != null && request.auth.token.email == 'admin@yourdomain.com'; } // Super Admin: list all users match /users/{userId} { allow list: if request.auth != null && request.auth.token.email == 'admin@yourdomain.com'; } } }

Resetting Your Data (Fresh Start)

To wipe all test data and start clean:

Firebase Console → Firestore Database
Navigate to the users collection and find your UID document.
Delete the data documents
Inside users/{your-uid}/data/ — delete meta, tracker, and diary. Also delete users/{uid}/profile/info.
Sign out and sign back in
The app detects no existing data and creates a fresh set with your correct owner schedule automatically.

Finding your UID: Firebase Console → Authentication → Users → see the UID column next to your email.

Bug Fixes Log
All confirmed bugs and their exact fixes

v14 Final — Latest Release

🐛 Admin tab disappeared after every save / interaction

Root cause: Inside saveS(), the function called on every habit tick, water click, diary keystroke, etc., there was a stray line: const _at=$id('admin-tab');if(_at)_at.style.display='none';

This hid the Admin tab on every single save operation.

✅ Fix
Removed the erroneous line from saveS(). Admin tab visibility is now managed only in initApp() (show for Super Admin) and sign-out (always hide).
🐛 Hide buttons (− hide) did nothing

Root cause: Buttons used onclick="collapseWidget('focus')" HTML attribute syntax. JavaScript <script type="module"> creates a private scope — functions inside it are NOT accessible via onclick attributes on HTML elements. The browser found collapseWidget as undefined on window and silently did nothing.

✅ Fix
Removed all onclick= attributes from HTML. Added proper listeners inside the module script: $id('focus-collapse-btn').addEventListener('click', ()=>collapseWidget('focus')); — same for water, focus-collapsed, water-collapsed.
🐛 Super Admin saw general schedule instead of Big Data / Finance

Root cause: DEF_WD(), applyOwnerSchedule(), and loadFromFirebase() all checked isOwner() only — which returns false for admin@yourdomain.com. The Super Admin account received the blank "Work / Study block" general template.

✅ Fix
Added isOwnerOrAdmin() helper and updated all 4 enforcement points to use it. Both accounts now receive the full 21-block owner schedule.
